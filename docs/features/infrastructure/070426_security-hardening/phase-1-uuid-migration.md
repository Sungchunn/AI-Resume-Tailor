# Phase 1: UUID Column Migration

**Parent:** `master-plan.md`
**Status:** Not Started

---

## Objective

Add `public_id` UUID columns to PostgreSQL tables that expose integer IDs in public APIs, without disrupting existing functionality.

---

## Scope

### Tables to Modify

| Table | Current State | Target State |
| ----- | ------------- | ------------ |
| `job_descriptions` | `id INTEGER PRIMARY KEY` | Add `public_id UUID UNIQUE NOT NULL` |
| `resume_builds` | `id INTEGER PRIMARY KEY` | Add `public_id UUID UNIQUE NOT NULL` |
| `user_job_interactions` | `id INTEGER PRIMARY KEY` | Add `public_id UUID UNIQUE NOT NULL` |

### Tables Excluded

| Table | Reason |
| ----- | ------ |
| `resumes` | Uses MongoDB ObjectId, no PostgreSQL integer exposure |
| `tailored_resumes` | Lookup via `mongo_id` (ObjectId string), not integer |
| `job_listings` | System-wide data, not user-owned |
| `users` | IDs never exposed in URLs |
| `ai_usage_records` | Internal only, no public API |
| `scraper_runs` | Admin-only |

## Edge Cases and Error Handling

### Large Table Performance

For tables with >100k rows, the UPDATE may lock the table. Use batched updates:

```sql
-- Batched UUID backfill for large tables
DO $$
DECLARE
    batch_size INT := 10000;
    affected INT;
BEGIN
    LOOP
        UPDATE job_descriptions
        SET public_id = gen_random_uuid()
        WHERE id IN (
            SELECT id FROM job_descriptions
            WHERE public_id IS NULL
            LIMIT batch_size
        );

        GET DIAGNOSTICS affected = ROW_COUNT;
        RAISE NOTICE 'Updated % rows', affected;

        EXIT WHEN affected = 0;

        -- Small delay to allow other queries
        PERFORM pg_sleep(0.1);
    END LOOP;
END $$;
```

### Concurrent Inserts During Migration

New rows inserted during the migration window will have `public_id = NULL` if the model hasn't been updated yet. The migration handles this with the `WHERE public_id IS NULL` clause.

**Mitigation:**

1. Update models FIRST (Step 1)
2. Deploy backend with new model code
3. Then run migration (which only affects existing NULL rows)

### UUID Collision

UUID v4 collision probability is negligible (1 in 2^122). However, the unique index provides protection:

- If collision occurs, INSERT fails with unique constraint violation
- Application can catch and retry with new UUID

## Rollback Procedure

If issues are discovered after migration:

```bash
# 1. Rollback migration
cd backend
alembic downgrade -1

# 2. Revert model changes (remove public_id from models)
git checkout -- app/models/job.py app/models/resume_build.py app/models/user_job_interaction.py

# 3. Restart backend
# The application will function normally without public_id
```

**Note:** Rollback is safe because:

- No existing columns are modified
- No foreign keys reference public_id yet
- API still uses integer IDs (not updated until Phase 2)
